# 香蕉Pro 图像生成异步化路线对比

**对比对象：** `v4.0` vs `A2-DO`  
**用途：** 方案评审 / 路线决策 / 会前快速同步  
**结论定位：** 一页决策版，不替代详细设计文档

---

## 一、结论先说

如果当前最优先目标是：

- **尽量不碰用户 API Key 安全边界**
- **最大化复用 NewAPI 已有异步轮询思路**
- **希望更保守、更容易先落地**

优先选 **v4.0**。

如果当前最优先目标是：

- **直接解决客户端弱网长连接问题**
- **不再借壳 Sora 视频协议**
- **希望未来围绕任务模型继续扩展**

优先选 **A2-DO**。

---

## 二、两条路线一句话定义

### v4.0

客户端先请求 `NewAPI`，`NewAPI` 把 `CF` 当作 Sora 风格异步上游来轮询，`CF Workflow` 负责真正执行生图。

### A2-DO

客户端先请求 `CF`，`CF` 立即返回 `task_id`，随后 `Workflow` 代用户直连 `NewAPI` 同步等待结果，客户端只轮询 `CF`。

---

## 三、核心差异总表

| 维度 | v4.0 | A2-DO | 判断 |
|---|---|---|---|
| 首次提交入口 | `NewAPI` | `CF` | A2-DO 更直接 |
| 后续轮询发起方 | `NewAPI` | 客户端 | A2-DO 更贴近任务模型 |
| 客户端长连接压力 | 仍部分依赖 `NewAPI` 链路 | 基本切掉 | A2-DO 更强 |
| `CF` 是否接触用户 API Key | 否 | 是，短期持有 | v4.0 更安全 |
| 计费所有权 | `NewAPI` | `NewAPI` | 两者相同 |
| `CF` 状态真相源 | `KV` 为主，需兜底 | `DO` 为主 | A2-DO 更稳 |
| 对外协议自然度 | 借壳 Sora 视频协议 | 自定义图片任务协议 | A2-DO 更自然 |
| 客户端改造成本 | 中等 | 中等偏高 | v4.0 略优 |
| `NewAPI` 改造压力 | 低 | 低 | 两者相近 |
| 长期演进性 | 一般 | 强 | A2-DO 更优 |
| 安全边界复杂度 | 中 | 高 | v4.0 更优 |
| 低概率不确定态 | 较低 | 存在 `CF -> NewAPI` 传输不确定态 | v4.0 略优 |

---

## 四、最关键的取舍

### 1. 安全面

**v4.0 优势明显。**

原因：

- `CF` 不需要持有用户自己的 `NewAPI API Key`
- 敏感信息只停留在客户端与 `NewAPI`
- `CF` 更像无状态协议转换层

**A2-DO 的代价：**

- `CF` 在任务执行期内必须短期持有用户 `API Key`
- 需要额外设计：
  - 加密保存
  - 不落明文日志
  - 短 TTL 清理
  - 任务归属校验

### 2. 客户端弱网体验

**A2-DO 优势明显。**

原因：

- 客户端不再承担 300 ～ 600 秒长连接
- 只需要：
  1. 提交一次
  2. 轮询状态
  3. 成功后拿结果

对移动端、脚本客户端、网页前台挂起场景都更友好。

### 3. 协议自然度

**A2-DO 优势明显。**

`v4.0` 的核心缺点不是不可用，而是“业务上是图片任务，协议上却借壳视频任务”。  
这会让：

- 接口理解成本更高
- 排障语义更绕
- 后续扩展受限

`A2-DO` 则直接把事情说清楚：这就是一个图片生成任务。

### 4. 一致性与状态可靠性

**A2-DO 优势明显。**

原因：

- `v4.0` 需要和 `KV` 最终一致性做长期博弈
- `A2-DO` 直接把 `DO` 作为权威状态源

这点对于“查询状态是否准确”尤其关键。

### 5. 不确定态风险

**v4.0 略优。**

`A2-DO` 的关键风险是：

- `Workflow` 已经把请求发给 `NewAPI`
- 但 `CF -> NewAPI` 的连接在读结果前断了
- 此时无法完全确认：
  - `NewAPI` 是否已经成功生图
  - 是否已经扣费
  - 是否应该重试

所以 A2-DO 设计里必须接受一个低概率错误码：

- `upstream_transport_uncertain`

而 `v4.0` 因为主要是 `NewAPI` 自己掌控异步轮询链路，这类不确定态更少。

---

## 五、复杂度对比

### v4.0 的复杂度主要在

- Sora 协议伪装
- `KV` 一致性兜底
- `Workflow` 生图逻辑
- `NewAPI` 轮询联调

### A2-DO 的复杂度主要在

- 任务入口协议
- `DO` 状态机
- 用户 API Key 加密与清理
- `CF -> NewAPI` 长连接异常边界
- 任务归属校验

### 哪条更容易首版落地

如果只看“短期最少新概念”：

- **v4.0 更容易**

如果看“以后少绕远路”：

- **A2-DO 更合理**

---

## 六、适用场景判断

### 更适合选 v4.0 的场景

1. 你现在最担心的是安全边界，而不是协议优雅度。
2. 你非常不希望 `CF` 短期持有用户 API Key。
3. 你希望第一版尽量复用 `NewAPI` 侧已有调度模型。
4. 你更接受“先保守上线，再慢慢演进”。

### 更适合选 A2-DO 的场景

1. 客户端弱网、掉线、长连接中断已经是当前最主要矛盾。
2. 你认为继续借壳 Sora 协议会把系统越做越绕。
3. 你愿意为更自然的长期架构，多承担一些一期安全设计成本。
4. 你后续确实可能继续做：
   - 任务取消
   - 历史任务
   - 推送通知
   - 网页端任务中心

---

## 七、推荐结论

### 推荐 v4.0，如果你当前优先级排序是：

1. 安全边界更稳
2. 首版上线更保守
3. 计费链路更少新风险
4. 客户端能接受继续部分依赖 `NewAPI` 协议形态

### 推荐 A2-DO，如果你当前优先级排序是：

1. 先把客户端弱网长连接问题正面解决
2. 协议语义要自然
3. 状态源要强一致
4. 后续围绕任务模型继续扩展

### 我的综合判断

如果这是一个**短期止血型改造**，优先 **v4.0**。  
如果这是一个**准备作为长期正式架构的异步方案**，优先 **A2-DO**。

---

## 八、推荐的决策方法

建议你评审时直接按下面 4 个问题拍板：

1. 是否接受 `CF` 在任务执行期短暂持有用户 `NewAPI API Key`？
   - 不能接受：选 `v4.0`
   - 能接受：两者都可继续评估

2. 当前最痛的，是不是客户端弱网长连接问题？
   - 是：A2-DO 权重明显上升
   - 不是：v4.0 更保守

3. 是否愿意继续长期维护 “图片任务借壳 Sora 协议”？
   - 不愿意：偏向 A2-DO
   - 能接受：v4.0 可继续

4. 这次改造是“止血版”还是“长期版”？
   - 止血版：偏向 v4.0
   - 长期版：偏向 A2-DO
