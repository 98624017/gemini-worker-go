# Rust 同步代理重构收益研究设计

**目标**

评估仓库根目录同步 Gemini 代理是否值得以 Rust 重构，并给出在
**不修改现有 Go 代码** 前提下的落地边界、收益判断与迁移策略。

本次设计只覆盖根目录同步代理，不覆盖 `async-gateway/`。

---

## 背景

当前仓库已经分化为两部分：

- 根目录同步代理：负责 Gemini 兼容入口、`inlineData` URL 拉取转
  base64、`output=url` 图片上传改写、`/proxy/image`、SSE 响应改写
- `async-gateway/`：负责异步任务受理、持久化、轮询与后台 worker

本次研究明确只评估根目录同步代理。用户关注点不是“语言统一”，而是
**性能与内存占用**，并且优先问题是**大图请求下的内存峰值**。

同时，用户要求：

1. Rust 重构项目必须放在仓库内独立新目录
2. 原有 Go 同步代理代码保持不动
3. 研究结果既要回答“值不值得重构”，也要说明“如果做，怎么迁”

---

## 研究范围

### 纳入评估

- `POST /v1beta/models/:model:generateContent`
- `POST /v1beta/models/:model:streamGenerateContent`
- `GET /proxy/image`
- 请求侧 `inlineData.data=http(s)` 抓取与 base64 回填
- 响应侧 `inlineData.data=base64` 上传与 URL 回填
- markdown 图片归一化
- 多图去重保留最大图
- R2 / legacy 上传链路
- admin 日志脱敏与基础统计

### 不纳入评估

- `async-gateway/`
- Cloudflare `r2-cleaner`
- NewAPI 账务、额度、鉴权体系

---

## 现状观察

当前同步代理已经做过一轮内存治理，但热点仍然明显：

1. 请求体先 `ReadAll` 再 `json.Unmarshal`，随后又重新编码发给上游
2. `inlineData URL -> base64` 需要先完整读入图片字节，再生成更大的
   base64 字符串
3. 非流式响应先整包 `ReadAll`，再解 JSON、改写、重新编码
4. `output=url` 时，又把 base64 解码回图片字节再上传，形成第二轮峰值
5. SSE 维护了接近 15MiB 的扫描缓冲池，说明大块流式响应已经是现实场景
6. 运行时已经依赖 `GOMEMLIMIT`、主动 GC、`FreeOSMemory` 缓解内存压力

这说明当前主要问题不是 HTTP 转发本身，而是**大字节对象在多轮处理中的
复制、膨胀与 GC 压力**。

---

## 方案对比

### 方案 A：继续保留 Go，只做内存专项治理

只优化热点路径，例如减少整包 `ReadAll`、改进 JSON 处理、把部分上传与
下载改成流式。

优点：

- 风险最低
- 交付最快
- 现有行为最稳定

缺点：

- 收益上限明显
- 仍受制于当前 `main.go` 结构和 Go 大对象回收特性
- 很难从根上消掉多轮字节复制

### 方案 B：Rust 替换热点链路，Go 保留外壳

将图片下载、base64 编解码、上传、SSE 处理拆为独立 Rust 服务或模块，
Go 继续保留入口与兼容层。

优点：

- 收益更聚焦
- 风险小于全量重写

缺点：

- 会形成双语言长期维护
- 服务边界、调试链路、部署复杂度都会上升
- 一旦后续继续演进，很容易出现责任边界模糊

### 方案 C：在仓库内新建独立 Rust 同步代理，完整平行重写

在仓库内新增独立目录，例如 `rust-sync-proxy/`，重写根目录同步代理全部
职责，保留原 Go 代码作为行为基线和回滚版本。

优点：

- 能针对整个数据面做系统性内存优化
- 边界清晰，不污染现有 Go 工程
- 便于做 Go/Rust 行为对照
- 后续切换通过部署替换完成，回滚简单

缺点：

- 初期验证成本最高
- 需要完整覆盖兼容行为

### 结论

采用 **方案 C**。

原因不是“Rust 普遍更快”，而是：对当前同步代理，真正的收益点集中在
图片与 base64 链路的内存峰值控制，而独立 Rust 实现最适合围绕这条主链路
重建数据面。

## 收益判断

### 预期收益来源

Rust 重构的收益主要来自以下几点：

1. 更容易把图片下载、上传、SSE 处理改成流式
2. 减少请求与响应整包驻留时间
3. 缩短大字节对象生命周期
4. 降低 base64 与原图字节同时共存的时间窗口
5. 避免 Go 在大对象高并发场景下的 GC 抖动与回收滞后

### 预期收益边界

以下判断基于代码结构推断，不是压测实测值：

- 如果线上图片请求占比较高，且 `output=url`、SSE 大图块较常见，
  Rust 版峰值 RSS 有机会下降 **30% 到 60%**
- 如果大多数只是普通文本转发，则收益会明显缩水，更可能落在
  **10% 到 25%**

### 不应夸大的收益

不应把本次重构收益描述成：

- “Rust HTTP 栈天然更高性能”
- “换语言就一定解决所有延迟问题”

更准确的表述应是：

> Rust 对本项目的主要价值，在于重建图片和 base64 处理链路，降低大对象
> 内存峰值与 GC 压力，而不是单纯替换语言语法。

---

## Rust 项目边界

Rust 重构项目必须放在仓库内独立目录，例如：

```text
rust-sync-proxy/
```

该目录包含：

- 独立 `Cargo.toml`
- 独立源代码与测试
- 独立 `Dockerfile`
- 行为对照所需的测试夹具

原有 Go 代码保持：

- 不改根目录 `main.go`
- 不改现有 Go 测试与行为
- 继续作为行为基线与回滚版本

---

## Rust 版模块设计

建议将 Rust 同步代理拆为以下模块：

### `http_entry`

负责路由、错误返回、请求分发、header 透传，不承载图片逻辑。

### `request_rewrite`

负责请求侧 JSON 扫描、`inlineData` URL 抓取、大小限制、MIME 识别、
base64 回填。

### `response_rewrite`

负责非流式 JSON 响应改写、markdown 图片归一化、多图去重、
`output=url` 上传回填。

### `stream_rewrite`

负责 SSE 增量解析与 chunk 改写，避免继续沿用大 Scanner 缓冲的粗放模式。

### `image_io`

负责图片下载、SSRF guard、超时与重试、外部抓取代理规则、legacy/R2 上传。

### `cache`

负责 L1 内存缓存、L2 磁盘缓存、后台下载桥接、TTL 与容量控制。

### `admin_observability`

负责管理页日志脱敏、cache hit、耗时统计、内存指标暴露。

## 内存优化原则

Rust 版必须围绕以下原则实现，否则语言切换收益会被浪费：

### 1. 能流式就不整包

- `/proxy/image` 保持纯流式转发
- SSE 使用增量 parser，不再依赖大 token 扫描
- 上传尽量支持边读边传，避免先完整落地第二份图片字节

### 2. JSON 选择性解析

- 不默认把所有请求和响应都解成完整通用对象树
- 只定位必要字段，如 `inlineData`、`parts`、`finishReason`
- 普通透传场景尽量走少改写路径

### 3. 缩短字节生命周期

- 图片字节在进入下一阶段后尽快释放
- base64 结果只在写回 JSON 的必要阶段短暂存在
- 避免原图字节、base64 文本、再解码字节长时间共存

### 4. 用内存预算控制并发

不仅限制并发任务数，也限制“总在途字节量”。这样对大图场景更稳，对小图
并发也更友好。

---

## 必须保持兼容的行为

Rust 版必须 1:1 保持以下用户可见语义：

- 路由与 query 语义不变
- Header 覆盖 `baseUrl|apiKey` 规则不变
- `output=url` 上传失败时保持 fail-open
- 请求侧 `inlineData URL -> base64` 失败时保持 fail-closed
- `ALLOWED_PROXY_DOMAINS`、`IMAGE_HOST_MODE`、R2 配置语义不变
- markdown 图片归一化行为不变
- 多张 `inlineData` 只保留 payload 最大图片的规则不变
- admin 日志仍然折叠 base64，避免敏感内容泄漏

---

## 错误处理策略

### 请求侧

- 图片下载失败、超时、超限时继续返回明确错误
- 后台抓取桥接语义保持一致

### 响应侧

- `output=url` 上传失败时只记录日志并保留 base64
- legacy 与 R2 回退策略保持现有 fail-open 思路

### 日志与安全

- URL 日志必须继续脱敏
- `/proxy/image` 的 allowlist 与 SSRF guard 不能弱化

---

## 验证与切换策略

虽然用户倾向“一次性重写”，但上线策略不应是一把切全量。

推荐采用 **单实现交付，双实现验收**：

1. Rust 独立目录完成实现
2. 使用现有 Go 行为作为对照基线
3. 先做离线输入集比对
4. 再做影子流量或旁路比对
5. 小流量切换
6. 指标稳定后再正式替换 Go 同步代理

### 验证层级

#### 单元测试

- allowlist / SSRF guard
- `output=url` 判定
- markdown 图片归一化
- 多图保留最大图
- MIME 推断与大小限制

#### 夹具测试

- 普通文本请求
- 单图 `inlineData`
- 多图异常响应
- markdown image 文本
- SSE 图像 chunk

#### Go/Rust 对照测试

同一组输入同时跑 Go 与 Rust，验证用户可见行为等价。

#### 压测

至少覆盖：

- 小请求高并发
- 大图请求中并发
- `output=url` + SSE 混合场景

核心指标：

- 峰值 RSS
- OOM 次数
- P95 / P99
- 单实例最大安全并发
- 上传失败回退比例

---

## 最终建议

对当前根目录同步代理，结论是：

1. **值得做 Rust 重构**
2. 但收益理由应聚焦在**图片与 base64 链路的内存峰值和 GC 压力**
3. Rust 项目应在仓库内以**独立目录**实现
4. 原 Go 代码保持不动，作为基线和回滚版本
5. 真正决定成败的不是换语言本身，而是是否落实流式、少复制、短生命周期
   的数据面设计

---

## 后续实现规划要求

后续实现计划必须满足：

1. 不修改现有 Go 同步代理代码
2. 在仓库内新增独立 Rust 目录
3. 先建立行为对照测试，再进入切流量准备
4. 以 TDD 和小步提交推进
